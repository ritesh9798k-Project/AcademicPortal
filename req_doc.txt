3. Page-by-Page Functional Specification
This section documents each page's purpose, UI components, operations available, and the 
chatbot commands that map to those operations.
3.1  Login Page  (/Account/Login)
Route
/Account/Login
Access
Public — All roles before authentication
Controller
AccountController → Login(), ExternalLogin()
View
Login.cshtml — centered card layout with university branding
• Email/Student ID and password fields with show/hide toggle
• Role auto-detection from credentials — no need for role selector dropdown
• "Remember Me" checkbox for persistent session
• Forgot Password link → /Account/ForgotPassword page
• After login, user is redirected to their role-specific dashboard automatically
• Chatbot command on login page: "Login" or "Sign in" — chatbot assists with credentials 
guidance

3.11  Dedicated Chatbot Interface Page  (/Chatbot/Index)
Route
/Chatbot/Index
Access
All Roles
Controller
ChatbotController → Index(), SendMessage(), GetHistory(), 
ExecuteAction()
View
Chatbot/Index.cshtml — full-page chat UI with sidebar
The chatbot also exists as a floating widget (bottom-right) on EVERY other page. The 
/Chatbot/Index page is the full-screen, full-feature version with history, session management, 
and a command palette.
• Full-page chat window with conversation history on left sidebar
• Message input with voice input support (Web Speech API)
• Command palette: click "/" to see all available chatbot commands
• Intent display badge on each bot response showing detected intent and confidence %
• Action confirmation cards: when chatbot is about to perform a write operation (enroll, book, 
cancel), it shows a confirmation card before executing
• Navigation actions: chatbot can push the browser to a new page using window.location
• File upload: student can upload a document and chatbot summarizes it or routes to advisor
• Session history: past conversations stored and searchable
• Thumbs up/down rating per bot response for training data collection

7. Role-Based Access Control
Every controller action is protected with [Authorize(Roles="...")] attributes. The shared navigation 
layout (_Layout.cshtml) uses @User.IsInRole("Student") conditions to show/hide menu items. The 
chatbot also enforces role restrictions — admin-only commands are rejected with an appropriate 
message if issued by a non-admin user.
